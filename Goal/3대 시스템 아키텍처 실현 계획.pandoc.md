### 2-1. TideCore.jar

#### 조수(Tide) 스케줄러

    [구현 목표]
      조수 상태를 주기적으로 전환하고, 그 상태를 서버 전체에 브로드캐스트한다.

    [핵심 클래스]
      TideScheduler
        - 현재 TideState (HIGH / LOW / SPRING / BLOOD_MOON / BLOOD_TIDE) 보관
        - BukkitRunnable으로 매 tick 남은 시간 카운트다운
        - 상태 전환 시 TideChangeEvent (커스텀 이벤트) 발행
        - BossBar에 "🌊 밀물 — 다음 변동까지 HH:MM:SS" 실시간 업데이트

    [TideState Enum]
      HIGH_TIDE      // 밀물
      LOW_TIDE       // 썰물
      SPRING_TIDE    // 사리 (~5% 확률 or 주 1회 예약)
      BLOOD_MOON     // 블러드문 (썰물 야간 저확률)
      BLOOD_TIDE     // 블러드 사리 (사리 + 블러드문 동시)

    [설정값 — config.yml 에서 핫 리로드 가능]
      tide.cycle-duration-minutes: 120   # 기본 밀물/썰물 한 사이클
      tide.spring-tide-chance: 5         # 사리 발생 확률 (%)
      tide.blood-moon-chance: 8          # 블러드문 발생 확률 (%)
      tide.scheduled-spring-day: SUNDAY  # 주 1회 예약 사리 요일

    [상태별 효과 적용 방식]
      TideChangeEvent 수신 → 각 파트의 리스너가 자기 책임 범위만 처리
      예) TideRPG → 드롭률 배율 변경
          TideMobs → 몹 스폰률·스탯 배율 변경
          TideCore → 플레이어 이동속도 포션 효과 부여
      → 파트 간 직접 호출 없이 이벤트로만 통신 (느슨한 결합)

#### 이중 화폐 & 경제 시스템

    [데이터 구조]
      PlayerEconomy (UUID → 데이터)
        clam: Long    // 조개 (Tide Coin)
        pearl: Long   // 진주 (Pearl)
        rep: Int      // 평판 (Reputation)

    [저장 방식]
      저장: SQLite (비동기 Async I/O, 메인 스레드 부하 0)
      파일: /plugins/TideCore/data/economy.db
      테이블: player_economy (uuid TEXT, clam INTEGER, pearl INTEGER, rep INTEGER)

    [API — 타 Jar가 호출하는 인터페이스]
      EconomyAPI.getClam(UUID)   → Long
      EconomyAPI.addClam(UUID, amount)
      EconomyAPI.takeClam(UUID, amount) → Boolean  // 잔액 부족 시 false
      EconomyAPI.getPearl(UUID)  → Long
      EconomyAPI.addPearl(UUID, amount)
      EconomyAPI.takePearl(UUID, amount) → Boolean
      EconomyAPI.addRep(UUID, amount)
      EconomyAPI.getRepTier(UUID) → RepTier (BRONZE / SILVER / GOLD / TIDE_MASTER)

    [인플레이션 방지 모니터링]
      매 10분마다 서버 전체 clam 총량 계산 → config 임계값 초과 시
      /tide admin 화면에 경고 표시 (자동 개입 없음, 운영자 판단)

#### 소프트 하드코어 죽음 --- "조류에 휩쓸림"

    [트리거]
      PlayerDeathEvent

    [처리 순서]
      1. 소지 clam의 10% 차감 (EconomyAPI.takeClam)
      2. 사망 위치에 WreckageGrave 생성
           - ArmorStand 소환 (CustomName: "§b[유실물 비석] §7남은 시간: 10:00")
           - 드롭 아이템을 HashSet<ItemStack>으로 비석 UUID에 매핑하여 메모리 보관
           - 바닐라 아이템 드롭 이벤트 cancel()
      3. 플레이어 부활 후 약화 디버프 부여
           - Weakness II / 60초
           - 스코어보드에 비석 좌표 + 카운트다운 표시
      4. BukkitRunnable (1분 주기) → 남은 시간 감소, ArmorStand 이름 갱신
      5. 10분 경과 시 ArmorStand 제거, 아이템 맵에서 삭제

    [회수 처리]
      PlayerInteractAtEntityEvent → 대상이 비석 ArmorStand인지 확인
      → 본인 UUID 비석인 경우: 아이템 인벤토리 복원, 비석 제거
      → 타인 비석: 메시지만 출력 ("다른 모험가의 유산입니다.")

    [하드 모드 토글]
      PlayerProfile에 hard_mode: Boolean 저장
      hard_mode=true 인 경우: 추가로 장비 최고 강화치 -1

### 2-2. TideRPG.jar

#### PDC 기반 장비 스탯 시스템

    [핵심 원칙]
      모든 RPG 수치는 PersistentDataContainer에 저장.
      Lore는 PDC 값을 읽어 렌더링만 함. 절대 Lore 문자열을 파싱하지 않음.

    [PDC 네임스페이스]
      NamespacedKey("tiде", "gs")          → PersistentDataType.INTEGER  // Gear Score
      NamespacedKey("tide", "socket_count")→ INTEGER   // 소켓 수 (0~3)
      NamespacedKey("tide", "socket_1")    → STRING    // "lifesteal:2"
      NamespacedKey("tide", "socket_2")    → STRING
      NamespacedKey("tide", "socket_3")    → STRING
      NamespacedKey("tide", "reinforce")   → INTEGER   // 강화 단계 +0~+10
      NamespacedKey("tide", "item_id")     → STRING    // "iron_sword_t1" 등 고유 ID
      NamespacedKey("tide", "cmd")         → INTEGER   // CustomModelData

    [장비 생성 흐름]
      ItemFactory.create("iron_sword_t1")
        1. items/ 디렉토리에서 iron_sword_t1.yml 로드
        2. ItemStack 기본 설정 (material, name, cmd)
        3. PDC에 gs, socket_count, item_id, reinforce(=0) 기록
        4. Lore 렌더링: LoreRenderer.render(itemStack) 호출
           → PDC 읽어서 § 컬러 코드로 Lore 생성 (표시 전용)
        5. 완성된 ItemStack 반환

    [Lore 렌더링 예시 출력]
      §7아이템 ID: §firon_sword_t1
      §6전투력(GS): §f150
      §7강화: §a+0
      §7소켓: §f[흡혈의 룬 II] [번개의 룬 I] [비어있음]

#### 스타포스 강화 시스템

    [강화 확률 테이블 — config에서 핫 리로드 가능]
      +1:  100%   실패 시 → 유지
      +2:   90%   실패 시 → 유지
      +3:   80%   실패 시 → +2로 하락
      +4:   70%   실패 시 → +3으로 하락
      +5:   60%   실패 시 → +4로 하락
      +6:   50%   실패 시 → +5로 하락
      +7:   40%   실패 시 → +6으로 하락
      +8:   35%   실패 시 → +7으로 하락
      +9:   30%   실패 시 → +8로 하락
      +10:  25%   실패 시 → +9로 하락 (보호권 사용 시 유지)
      * 파괴 없음 (rage quit 방지 설계)

    [강화 GUI 구현]
      /forge 명령어 → InventoryView 열기 (크기: 9칸)
      슬롯 4: 강화할 아이템 배치
      슬롯 2: 강화석 (재료 — item_id="reinforce_stone")
      슬롯 6: 보호권 (선택 — item_id="protection_scroll", 진주 소모)
      슬롯 8: [강화 시도] 버튼 (에메랄드)

      InventoryClickEvent → 슬롯 8 클릭 시:
        1. 슬롯 4 아이템 PDC "reinforce" 값 읽기 → 현재 단계 확인
        2. 재료(강화석) 보유 확인
        3. 보호권 사용 여부 확인
        4. 조개 비용 차감 (단계별 차등)
        5. Math.random() < 확률 → 성공/실패 분기
        6. PDC "reinforce" 값 업데이트
        7. LoreRenderer.render() 재호출로 Lore 갱신
        8. 성공/실패 파티클·사운드 출력

    [강화에 따른 스탯 보너스]
      base_damage + (reinforce_level * damage_per_star)
      damage_per_star 값은 아이템 yml의 reinforce_bonus 필드에서 로드

#### 전투 리스너 (룬 효과 발동)

    [EntityDamageByEntityEvent 처리 흐름]

      이벤트 수신
        ↓
      공격자가 Player 인지 확인
        ↓
      공격자 메인핸드 아이템 PDC 읽기
        socket_1, socket_2, socket_3 순회
        ↓
      각 소켓 값 파싱: "lifesteal:2" → type=lifesteal, grade=2
        ↓
      RuneEffectDispatcher.dispatch(type, grade, attacker, victim, damage)

    [룬 효과별 구현]

      lifesteal (흡혈의 룬):
        회복량 = finalDamage * (0.08 * grade)
        attacker.setHealth(min(maxHp, currentHp + 회복량))

      lightning (번개의 룬):
        발동확률 = 0.10 * grade
        if (Math.random() < 발동확률):
          victim.getWorld().strikeLightningEffect(victim.getLocation())
          victim에게 추가 데미지 적용

      slow (둔화의 룬):
        지속시간 = 40 + (grade * 20) ticks
        victim.addPotionEffect(PotionEffect(SLOW, 지속시간, grade - 1))

      shield (방벽의 룬):  // 피격 시 발동
        → EntityDamageEvent 에서 별도 처리
        흡수막 = 5.0 * grade
        attacker.setAbsorptionAmount(흡수막)

      berserk (광폭의 룬, 저주):
        → DamageModifier로 +30% 데미지 보너스
        → EntityDamageEvent에서 received damage +15% 적용

### 2-3. TideMobs.jar

#### 접두사(Affix) 정예 몹 시스템

    [CreatureSpawnEvent 처리 흐름]

      자연 스폰 이벤트 수신
        ↓
      현재 TideState 확인 → 정예 출현 확률 결정
        HIGH_TIDE:   5%
        LOW_TIDE:    12%
        SPRING_TIDE: 20%
        BLOOD_MOON:  25%
        ↓
      Math.random() < 확률 → 정예화 진행
        ↓
      AffixRegistry에서 랜덤 접두사 1~2개 선택
        (몹 종류별 허용 affix 목록 yml에서 로드)
        ↓
      EliteProcessor.apply(entity, affixes[])
        1. 체력 배율 적용 (affix별 hp_multiplier 합산)
        2. 데미지 배율 PDC에 기록
        3. 이름 설정: "§c[§6염화의§c] §f좀비"
        4. 발광 효과 적용 (Glowing)
        5. 크기 조정 (일부 affix)
        6. affix 리스트를 PDC에 직렬화하여 저장
           NamespacedKey("tide", "affixes") → STRING ("염화의,신속의")
        7. 드롭 테이블 재설정 (정예 전용 드롭 추가)

    [접두사별 전투 동작 — EntityDamageByEntityEvent에서 처리]

      [염화의]:
        피격 시 피격 대상 주변 3×3에 FIRE 블록 2초 생성
        발동확률: 30%

      [신속의]:
        평소 이동속도 +50%, 10% 확률로 순간이동(3블록 앞으로)

      [폭심의]:
        사망 시(EntityDeathEvent): 위치에 Explosion(power=3, fire=false)

      [분열의]:
        HP 50% 이하 시(EntityDamageEvent): 동일 몹 1마리 추가 스폰
        분열 횟수 PDC로 추적 (최대 2회)

      [강철의]:
        수신 데미지 * 0.6 (40% 감소)

      [재생의]:
        20틱(1초)마다 최대 HP의 2% 회복
        → EntityTickEvent 대신 스폰 시 BukkitRunnable 등록

      [가시의]:
        수신 데미지의 30%를 공격자에게 반사

      [보호막의]:
        스폰 시 HP의 20%에 해당하는 흡수막 부여
        흡수막 소멸 시 15초 쿨다운 후 재생성

#### 네메시스(Nemesis) 시스템

    [데이터 구조]
      NemesisRecord
        mob_uuid: UUID         // 네메시스가 된 정예 몹 UUID
        player_uuid: UUID      // 복수 대상 플레이어 UUID
        original_name: String  // 원래 이름 ("좀비")
        affixes: List<String>  // 원래 보유 접두사
        kill_count: Int        // 해당 플레이어를 죽인 횟수
        is_active: Boolean     // 현재 활성 상태

      저장: SQLite nemesis 테이블
      (서버 재시작 후에도 네메시스 유지)

    [각성 트리거 1 — 플레이어 사망]
      PlayerDeathEvent

        4. 로드 결과 (성공 N건, 실패 M건) 콘솔 출력

    [Python 스크립트 — generate_content.py]

    import anthropic
    import yaml
    import os
    import sys

    def generate_mob(user_request: str) -> str:
        client = anthropic.Anthropic()

        with open("prompts/system_mob.txt", "r") as f:
            system_prompt = f.read()

        message = client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=2000,
            system=system_prompt,
            messages=[{"role": "user", "content": user_request}]
        )

        yaml_text = message.content[0].text

        # 유효성 검사
        try:
            data = yaml.safe_load(yaml_text)
            required_keys = ["id", "base_mob", "stats", "affixes", "drops"]
            for key in required_keys:
                assert key in data, f"필수 필드 누락: {key}"
        except Exception as e:
            print(f"[오류] YAML 유효성 검사 실패: {e}")
            return None

        # 파일 저장
        output_path = f"output/mobs/{data['id']}.yml"
        with open(output_path, "w", encoding="utf-8") as f:
            f.write(yaml_text)

        print(f"[완료] 파일 생성: {output_path}")
        return output_path

    if __name__ == "__main__":
        request = " ".join(sys.argv[1:])
        generate_mob(request)

### 5-2. GS 기반 자동 가이드라인

    [트리거 1 — 지역 진입]
      PlayerMoveEvent (청크 경계 체크로 최적화)
        → 플레이어 현재 위치가 GS 게이트 구역에 진입했는지 확인
        → zone_config.yml 에서 구역별 recommended_gs, warn_gs 로드

      recommended_gs: 400   # 이 GS 미만이면 노랑 경고
      warn_gs: 250          # 이 GS 미만이면 빨강 경고

      처리:
        playerGS = 플레이어 착용 장비 PDC의 gs 합산
        if playerGS < warn_gs:
          title("§c⚠ 위험", "§f현재 GS: " + playerGS + " / 권장: " + recommended_gs)
        elif playerGS < recommended_gs:
          title("§e주의", "§f현재 GS: " + playerGS + " / 권장: " + recommended_gs)

    [트리거 2 — 보스 제단 접근]
      PlayerInteractEvent → 상호작용 블록이 보스 제단인지 PDC 확인
      → 동일 GS 비교 로직 적용
      → 추가로 "권장 파티 인원: N명" 안내

    [플레이어 GS 계산]
      GearScoreCalculator.calculate(player)
        → 투구·흉갑·레깅스·부츠·주무기·보조 장비 PDC gs 합산
        → 강화 단계 보너스 포함: gs + (reinforce * gs_per_star)

## 6. 3-Jar 모듈 분리 구조

    [의존 관계]
      TideCore.jar     ← 독립 (의존 없음)
      TideRPG.jar      ← TideCore.jar API 사용 (EconomyAPI)
      TideMobs.jar     ← TideCore.jar API 사용 (EconomyAPI, TideState)
                       ← TideRPG.jar API 사용 (GearScoreCalculator)

    [파일 구조]
      /plugins/
        TideCore/
          config.yml
          data/
            economy.db
            nemesis.db
            graves.db
        TideRPG/
          items/           ← 아이템 YAML들 (3파트가 생성)
          runes/           ← 룬 YAML들
          forge_config.yml
        TideMobs/
          mobs/            ← 몹 YAML들 (3파트가 생성)
          affixes/
            염화의.yml
            신속의.yml
            ...
          nemesis_config.yml

    [Jar 간 통신 방식]
      직접 클래스 참조가 아닌 Bukkit ServiceManager 등록
      TideCore → ServicesManager.register(EconomyAPI.class, impl, plugin, NORMAL)
      TideRPG  → ServicesManager.getRegistration(EconomyAPI.class).getProvider()
      → 로드 순서: TideCore → TideRPG → TideMobs (plugin.yml depend 설정)

## 8. 개발 단계별 구현 순서

    [0단계 — 토대 (1~2일)]
      목표: 서버 구동 + 기반 데이터 확보

      □ Paper 서버 세팅 (최신 LTS)
      □ TideCore.jar 뼈대 생성
          □ plugin.yml (name, version, api-version)
          □ EconomyManager (SQLite 연결, 플레이어 프로필 CRUD)
          □ EconomyAPI (ServicesManager 등록)
          □ 기본 config.yml 로더
      □ /clam, /pearl 확인 명령어 (디버그용)
      □ 테스트: 플레이어 접속 시 프로필 자동 생성 확인

    [1단계 — 핵심 재미 (3~5일)]
      목표: 여기서 게임이 재밌어짐 — 도파민 루프 개시

      □ TideRPG.jar 기본 구조
          □ ItemFactory (YAML → ItemStack 생성)
          □ LoreRenderer (PDC → Lore 텍스트 렌더링)
          □ 샘플 아이템 YAML 3~5개 직접 작성 (나중에 AI가 생성)
      □ TideMobs.jar 기본 구조
          □ AffixRegistry (접두사 yml 로드)
          □ CreatureSpawnEvent 리스너 → 정예화 로직
          □ 접두사 전투 효과 (염화의 먼저 구현)
      □ RuneEffectDispatcher (lifesteal, lightning 먼저)
      □ 기본 상점 GUI (hardcode된 아이템 몇 개, 조개로 구매)
      □ 테스트: 야생에서 정예 몹 스폰 + 룬 효과 발동 확인

    [2단계 — 성장 (3~4일)]
      목표: 강해지는 느낌을 숫자로 주기

      □ 스타포스 강화 GUI (/forge 강화 탭)

          □ 특정 영역 블록 주기 리셋 (WorldEdit API 활용 또는 직접 구현)
          □ 딥 마인 사망 추방 + 전리품 일부 소실
      □ GearScoreCalculator

      □ 테스트: +1~+5 강화 사이클 완료, GS 표시 확인

## 9. AI Agent 지시 템플릿

> 개발 시 AI Agent(Claude Code 등)에게 각 모듈을 맡길 때 사용하는 표준
> 지시문.

### 9-1. TideCore 스케줄러 지시문

    [지시]
    Paper API(최신 LTS 호환)로 TideScheduler 클래스를 작성해줘.

    [요구사항]
    - TideState Enum: HIGH_TIDE, LOW_TIDE, SPRING_TIDE, BLOOD_MOON, BLOOD_TIDE
    - BukkitRunnable으로 1틱마다 카운트다운, 상태 전환 시 TideChangeEvent 발행
    - BossBar(BarColor.BLUE, BarStyle.SOLID)에 "🌊 밀물 — 다음 변동까지 HH:MM:SS" 표시
    - 사리 확률: config에서 읽은 spring_tide_chance (%)
    - 블러드문: 썰물 + 야간(isNight) + blood_moon_chance (%)
    - 블러드 사리: 사리와 블러드문이 동시에 트리거되는 경우
    - config.yml에서 cycle-duration-minutes 핫 리로드 가능

    [금지]
    - 매 틱 전체 엔티티 스캔 금지
    - 정적 필드 사용 최소화 (싱글톤은 plugin 인스턴스를 통해서만)

### 9-2. PDC 기반 아이템 시스템 지시문

    [지시]
    Paper API로 ItemFactory와 LoreRenderer 클래스를 작성해줘.

    [요구사항]
    - ItemFactory.create(String itemId): items/ 디렉토리에서 yml 로드 → ItemStack 반환
    - PDC 키 네임스페이스: "tide"
    - 저장 키: item_id(STRING), gs(INTEGER), reinforce(INTEGER, 기본값 0),
               socket_count(INTEGER), socket_1~3(STRING), cmd(INTEGER)
    - LoreRenderer.render(ItemStack): PDC 읽어서 Lore 재생성 (§ 컬러 코드 사용)
    - ItemRegistry: 서버 시작 시 items/ 전체 로드, reload() 메서드 제공

    [절대 금지]
    - Lore 문자열 파싱으로 수치를 읽는 코드
    - ItemMeta의 lore에서 수치를 추출하는 코드

### 9-3. 정예 몹 시스템 지시문

    [지시]
    Paper API로 EliteProcessor와 AffixRegistry를 작성해줘.

    [요구사항]
    - CreatureSpawnEvent에서 SpawnReason이 자연 스폰인 경우만 처리
    - 현재 TideState에 따라 정예 출현 확률 다르게 적용
      (TideCore의 ServicesManager에서 TideStateProvider 인터페이스로 가져옴)
    - AffixRegistry: affixes/ 디렉토리의 yml 로드
      각 yml에 name, hp_multiplier, damage_multiplier, spawn_particle 등 정의
    - EliteProcessor.apply(LivingEntity, List<Affix>):
      체력 배율 적용, 이름 변경, 발광, PDC에 "tide:affixes" 기록
    - 성능: 스폰 시 1회만 처리, 이후 tick 루프 없음

    [금지]
    - runTaskTimer로 매 틱 엔티티 목록 순회
    - 전역 static Map에 엔티티 보관 (메모리 누수 위험)
      → WeakHashMap 사용 또는 PDC만 활용
